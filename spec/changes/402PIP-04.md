# 402PIP-04: provider self-registration

**status:** draft (target v0.2)
**author:** mikrohash
**created:** 2026-05-04

## summary

remove the hard-coded provider catalog. let any provider register itself on-chain via a small registry program, paying a refundable sol deposit as anti-spam.

## motivation

v0.1 ships a fixed list of six providers in `spec/providers.md`. that does not scale:

- new providers need a maintainer PR
- maintainers become a chokepoint
- the protocol's "open" framing is undermined by a curated list

self-registration moves the catalog on-chain and lets clients discover providers via a single rpc query.

## proposal

### on-chain registry

a solana program at a known address holds the provider records.

- **register**: `register(id, category, base_url, methods, signer_pubkey, treasury_pubkey, deposit_lamports)` — writes a record, locks a deposit.
- **deregister**: `deregister(id)` — returns the deposit if the provider passed the quality bar.
- **update**: `update(id, ...)` — changes mutable fields (base_url, treasury_pubkey).

### deposit

```
deposit = 0.5 SOL
refund condition = >= 50 successful settles in 30 days
```

deposits in good standing for >30 days unlock automatic refund-on-deregister. abusive providers lose the deposit on a manual maintainer flag.

### client discovery

```
$ 402proto providers list
... [reads from registry, caches 60s]
```

clients pull the catalog via `getProgramAccounts` and cache locally.

### envelope impact

zero. the wire format does not change. only the discovery flow moves.

## rationale

- **anti-spam via deposit, not curation**: keeps the "open" framing.
- **client caching**: rpc cost is paid once per cache window, not per call.
- **mutable base_url**: providers move infrastructure all the time. record-level update without re-registering.

## open questions

- should `category` be a fixed enum or freeform string? leaning freeform with a "well-known categories" list maintained in spec, but no enforcement.
- registry program upgrade path: should we ship it as a freeze-after-launch contract or behind a multisig?
- spam: 0.5 SOL might be too low for active mainnet. open to adjustment.

## timeline

target: merged into v0.2. spec text will land in `spec/providers.md` once contract is audited and deployed on devnet.
