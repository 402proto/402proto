# 402PIP-05: revocable quotes

**status:** draft (target v0.2)
**author:** 0xnova
**created:** 2026-05-05

## summary

let providers revoke a quote before its ttl expires by publishing a small on-chain signal. clients re-quote on detection.

## motivation

current v0.1 quotes are valid for `ttl` seconds unconditionally. there are two scenarios this hurts:

1. **price moves**: an oracle quote at $0.001 may become unprofitable for the provider 5 seconds later if a backing service rates spike. provider has to choose between honoring a stale price or rejecting the settle and burning the agent's trust.
2. **rate limit reached**: provider hits a downstream rate limit between issuing and settling. current spec says provider MUST honor; this is operationally painful.

revocable quotes let providers signal "this quote is no longer valid" without lying about ttl globally.

## proposal

### new on-chain mechanism

each provider holds an on-chain `revocations` account. revoking a quote = appending its `quote_id` hash to the account.

clients querying settle time check the revocations account; if the quote_id is present, the client gets a `quote-revoked` error and must re-quote.

### envelope changes

minor — new error code:

```
quote-revoked  410  re-quote  quote was withdrawn by the provider before settle
```

added to `spec/errors.md`. no header changes.

### provider-side cost

writing to a solana account costs ~$0.0001 per revoke. providers SHOULD set their `revocations` account to a write-cheap layout (append-only buffer with circular eviction).

### incentives

providers that revoke too often get downscored by routers (similar to existing provider-error scoring). clients SHOULD prefer providers with low revocation rates.

## rationale

- **on-chain signal not off-chain**: keeps verification trustless. an off-chain "this quote is revoked" can't be trusted by the agent's wallet.
- **opt-in per provider**: providers that never need to revoke can ignore the mechanism. their `revocations` account stays empty.
- **append-only**: prevents replay games where a provider "un-revokes" a quote after the price moves back.

## open questions

- account size cap: how long should we keep revocations queryable? proposing 24h rolling window.
- gas cost on the client side: every settle now requires one extra rpc query. acceptable?

## timeline

target: v0.2. blocked on 402PIP-04 (registry) for the revocations-account discovery flow.
