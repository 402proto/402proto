# 402PIP-02: ed25519 signing over canonical body

**status:** accepted (v0.1)
**author:** mikrohash
**created:** 2026-04-16

## summary

every settled call carries an ed25519 signature from the agent's wallet over a deterministic canonical body. the signature is the proof-of-authorization the provider checks before serving the paid response.

## motivation

we need a signing scheme that:

- uses the same key the agent already has on solana (no new keypair to manage)
- is verifiable by any party — provider, router, future auditor — without solana rpc access
- prevents tampering with the http transport between quote and settle

ed25519 over a canonical body string is the obvious answer: solana wallets are ed25519, libraries exist everywhere, and a stable canonical body lets verification be a pure-function check.

## proposal

see [spec/signing.md](../signing.md). canonical body format:

```
402proto/0.1
provider=<...>
method=<...>
price=<...>
currency=<...>
recipient=<...>
chain=<...>
quote_id=<...>
nonce=<...>
wallet=<...>
ttl_expires_at=<unix seconds>
```

11 lines, `\n` separator, no trailing newline. base58-encoded signature transported in `x-402proto-signature`.

## rationale

- **canonical body over headers**: providers and clients sometimes normalize headers (case, ordering). a separate canonical body is immune to that.
- **ttl_expires_at in the signed body**: makes the signature itself bind to a specific expiry, so a stolen signature cannot be replayed after ttl.
- **explicit wallet field**: lets routers / verifiers check "the signer matches the claimed wallet" without trusting the header alone.
- **plain ed25519, not jose/jwt**: jose is heavy and the protocol crowd is more comfortable hand-rolling a sign-over-bytes scheme. test vectors are short and easy to cross-implement.

## acceptance

merged into `spec/signing.md` as v0.1. all v0.1 clients sign; all v0.1 providers verify.
