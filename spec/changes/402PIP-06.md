# 402PIP-06: eip-3009 settle on l2

**status:** discussion (target v0.4)
**author:** 402proto
**created:** 2026-05-08

## summary

extend 402proto settle to evm L2s (base, arbitrum, optimism) using usdc's `transferWithAuthorization` (eip-3009). the protocol stays envelope-compatible; only the settle mechanism differs per chain.

## motivation

solana is fastest but not where every agent runs. fleets pushing claude through anthropic's api may already hold usdc on base via coinbase wallet. forcing them to bridge to solana adds friction.

eip-3009 is the closest evm primitive to a memo-tagged spl-transfer:

- one signature
- recipient + amount baked into the signature
- no contract approval flow
- usdc supports it natively on base / arb / op

## proposal

### chain identifier

`x-402proto-chain` extends to:

```
solana-mainnet
solana-devnet
base
arbitrum
optimism
```

### settle for evm chains

instead of an spl-transfer + memo, the client constructs an eip-712 `transferWithAuthorization` payload:

```
TransferWithAuthorization(
  from: <agent wallet>,
  to: <x-402proto-recipient>,
  value: <x-402proto-price * 10^6>,
  validAfter: <now>,
  validBefore: <x-402proto-expires>,
  nonce: keccak256(x-402proto-quote-id)
)
```

signs eip-712 and sends the signature + the same envelope headers to the provider in the settle request. provider submits the `transferWithAuthorization` on its own, paying its own gas (huge UX win over the current solana flow).

### envelope impact

minimal:

- `x-402proto-receipt` carries an evm tx hash (0x...) for evm chains
- `x-402proto-signature` carries an eip-712 signature instead of an ed25519 signature for evm chains
- `x-402proto-wallet` carries a checksummed ethereum address for evm chains

### canonical body

the v0.1 canonical body is solana-specific (ed25519 sig over plain bytes). for evm chains, the canonical body is the eip-712 typed-data struct.

we add a `chain_family` field to the canonical body:

```
402proto/0.4
chain_family=solana   # or "evm"
...
```

solana clients keep the v0.1 body. evm clients use the eip-712 hash.

## rationale

- **provider pays gas**: only viable on l2 (mainnet gas would kill the model). this is the entire reason to do this on l2 and not eth mainnet.
- **eip-3009 over permit**: permit requires an approve+transferFrom dance. eip-3009 is a single signed authorization, much closer in shape to solana's spl-transfer.
- **chain_family field**: avoids per-chain branching in core code. solana family stays binary-clean; evm family is eip-712.

## NOT in scope

- ethereum mainnet — gas too expensive. l2 only.
- non-evm non-solana chains — defer. cosmos, sui, etc. handle separately if there is real demand.
- bridge-based settle. clients bring usdc to the chain they want to pay on.

## open questions

- does provider-pays-gas need a separate gas-reimbursement mechanism, or do providers just price-in expected gas? leaning the latter — simpler, providers already do this for their compute.
- which l2 first? probably base, given coinbase's existing 402 traction and the wallet distribution.

target: v0.4. blocked on v0.2 (mainnet settle), v0.3 (fleet wallet abstractions).
